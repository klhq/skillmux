## Goal
Safely tune, persist, and activate model-specific routing decision thresholds while preserving reproducible calibration evidence and a last-known-good runtime configuration.

## Acceptance Criteria
- [x] **Acceptance Criterion 1 — Decision-policy dataset & Synthetic Generator:** Evaluation cases use the exact fields `query`, `split`, `expected_outcome`, and `relevant_skill_ids`; `split` is `tune` or `test`, `expected_outcome` is `matched`, `ambiguous`, or `no_match`. Checked-in `eval/queries.json` is sanitized using generic synthetic mock queries and skill IDs. Local synthetic generator (`generateDataset`) and CLI `skillmux calibrate generate-dataset` generate vault-tailored calibration datasets on demand.
- [x] **Acceptance Criterion 2 — Reproducible tuning run:** `skillmux calibrate run` requires a configured reranker but does not require pre-existing reranker thresholds. It runs hybrid retrieval and reranking once per evaluation query, caches the resulting candidate IDs, ranks, scores, and relevance labels, and searches only the cached observations when selecting `inference.thresholds.match_score`, `inference.thresholds.match_margin`, and `inference.thresholds.candidate_floor`.
- [x] **Acceptance Criterion 3 — Explicit optimization contract:** Threshold selection maximizes correct auto-match coverage on `split = "tune"` subject to configurable `--min-auto-match-precision` and `--min-shortlist-recall-at-5` gates, whose defaults are `0.99` and `0.95`. Candidate threshold values are derived from observed score and margin breakpoints, and ties are resolved deterministically by higher auto-match precision, then higher shortlist recall at 5, then lower auto-match coverage. A run that cannot satisfy both gates has status `failed_gates` and cannot be applied.
- [x] **Acceptance Criterion 4 — Untouched test report:** After selecting thresholds from the tuning split, the run evaluates them once on `split = "test"` and reports `auto_match_precision`, `auto_match_coverage`, `shortlist_recall_at_5`, `false_no_match_rate`, and a three-outcome confusion matrix. Candidate-generation recall is reported separately so failures that threshold tuning cannot repair are visible.
- [x] **Acceptance Criterion 5 — Calibration evidence in SQLite:** Every calibration run is stored under `state_dir` with a stable `run_id`, timestamps, status, reranker/model fingerprint, embedding fingerprint, vault/corpus fingerprint, dataset hash, optimization gates, selected thresholds, tuning metrics, test metrics, and cached per-query candidate scores. SQLite is evidence and history only; it is never consulted on the `resolve_skill` request path and never becomes the authoritative active configuration.
- [x] **Acceptance Criterion 6 — Inspectable and gated application:** `skillmux calibrate list` and `skillmux calibrate show <run_id>` expose stored runs. `skillmux calibrate apply <run_id>` rejects missing runs, `failed_gates` runs, and runs whose model, corpus, or dataset fingerprint no longer matches. A valid run atomically writes the three selected values under `inference.thresholds` and its provenance under `inference.calibration.run_id` to the resolved TOML file.
- [x] **Acceptance Criterion 7 — Single desired-state authority:** TOML remains the desired configuration, environment variables remain higher-precedence deployment overrides, and an immutable in-memory runtime snapshot remains the active configuration. Calibration never writes active thresholds only to SQLite. Any attempted write to a key masked by an environment override fails with a clear source-aware error rather than reporting a change that cannot become effective.
- [x] **Acceptance Criterion 8 — Transactional live reload:** `skillmux serve` watches the resolved configuration file's parent directory so change- and rename-based atomic saves are detected. After debouncing, it reads a stable candidate, parses TOML, validates the complete effective configuration, prepares any required runtime dependencies, and atomically swaps the immutable runtime snapshot only after all steps succeed.
- [x] **Acceptance Criterion 9 — Explicit live-reload boundary:** The only live-reloadable keys are `inference.thresholds.match_score`, `inference.thresholds.match_margin`, `inference.thresholds.candidate_floor`, `recall.k_lexical`, `recall.k_vector`, and `thresholds.candidate_limit`. A candidate changing any other effective key is retained as desired state but is not partially activated; runtime status reports the exact restart-required keys while the previous snapshot continues serving.
- [x] **Acceptance Criterion 10 — Last-known-good behavior:** Invalid TOML, schema validation failure, failed dependency preparation, watcher errors, and restart-required drift never crash the server or discard the active snapshot. Readiness remains based on the last-known-good runtime, while runtime status exposes `active_revision`, `active_source_hash`, `desired_source_hash`, `last_successful_reload_at`, `last_reload_error`, and `restart_required_keys`.
- [x] **Acceptance Criterion 11 — Safe lifecycle and concurrency:** Replacing a runtime snapshot closes superseded database/client resources only after in-flight requests release them, never leaks the previous database handle, and shuts down configuration and vault watchers cleanly. Concurrent requests observe either the complete old snapshot or the complete new snapshot, never a partially applied configuration.

## Scope
- Extend the evaluation-case schema and checked-in evaluation data to represent tune/test splits and the complete `matched`/`ambiguous`/`no_match` decision policy with generic synthetic mock queries, and add on-demand synthetic dataset generation.

- Implement pure, deterministic threshold optimization over cached reranker observations.
- Add the local `calibrate run`, `calibrate list`, `calibrate show <run_id>`, and `calibrate apply <run_id>` command services; CLI presentation polish is specified separately in `docs/sdd/cli-polish/spec.md`.
- Store calibration runs, metrics, fingerprints, and score evidence in SQLite under `state_dir`.
- Add atomic TOML threshold persistence with calibration provenance.
- Introduce an immutable runtime-config snapshot manager with last-known-good semantics, resource-safe swaps, status reporting, and a narrow live-reload allowlist.
- Add unit and integration coverage for dataset validation, optimization, fingerprint staleness, persistence, atomic writes, rename-based saves, invalid and partial writes, event bursts, concurrent requests, failed preparation, restart-required changes, rollback behavior, and watcher shutdown.

## Out of Scope
- Probability-calibrating reranker scores or changing the reranker/embedding models themselves.
- Treating SQLite as the authoritative configuration store or reading configuration from SQLite on the request path.
- Optimizing candidate-generation parameters beyond reporting candidate-generation recall.
- Automatically applying a run that fails its gates or whose fingerprints are stale.
- Remote target discovery, named contexts, remote administration endpoints, CLI styling, shell completions, or the overall CLI command hierarchy; those belong to `docs/sdd/cli-polish/spec.md`.
- Automatically restarting the server for restart-required configuration changes.

## Tasks
1. Deliver an in-memory `calibrate run` slice with the validated tune/test dataset, cached observations, deterministic optimizer, gate failure, and untouched-test report.
2. Make runs durable and inspectable by persisting fingerprints, metrics, and per-query evidence in SQLite and wiring `calibrate list` and `calibrate show <run_id>` to that history.
3. Deliver safe application by validating a stored run's status and fingerprints, atomically writing its thresholds and `inference.calibration.run_id`, and proving masked environment values are rejected.
4. Introduce acquired immutable runtime snapshots, route requests through them, and prove old resources remain valid for in-flight requests and close after release.
5. Complete live activation with parent-directory watching, stable candidate reads, the reload allowlist, last-known-good behavior, restart-required status, and clean watcher shutdown.
6. Exercise the complete run-to-apply-to-reload flow end to end, including bootstrap without existing thresholds, rename-based saves, concurrent requests, and failure retention.
